---
name: Ghostlight admin UI — two helper builders must stay in sync
description: Why an admin page can render in the standalone preview but break on the deployed full bot
---

The Ghostlight admin web UI is rendered by shared functions in `src/http/renderAdminPages/*`, but the `helpers` bag those functions destructure is built in **two independent places** that can drift:

- `preview.js` → `buildHelpers(theme)` — used by the standalone, no-DB preview (`node artifacts/ghostlight-bot/preview.js`).
- `src/http/adminRenderHelpers.js` → `buildAdminPageHelpers(...)` — used by the **full bot** (`createHealthServer` → `handleAdminPageRequest`).

**Rule:** any helper key a page render function reads off `helpers` must be present in BOTH builders, and every full-bot page handler must actually pass `helpers` down to the raw render function (some `buildAdminPageHelpers` entries are the *raw* imported render fns, not wrappers, so they need `helpers` passed at the call site).

**Why:** the Companion page (`renderCompanionPage` → `getRuntimeState`) reads `helpers.extractRuntimeSettings` and `helpers.escapeHtml`. The preview's `buildHelpers` had them, so the page looked fine in the workspace preview, but `buildAdminPageHelpers` omitted `extractRuntimeSettings` and the companion/behaviour handlers didn't pass `helpers` at all — so the page rendered blank ONLY on the deployed full bot. Classic "works in preview, blank on Railway" trap.

**Extra trap — per-page wrapper helper bags in `adminRenderHelpers.js`:** many render fns are exposed via small wrapper fns (e.g. `renderCompanionPage(params)`) that **rebuild a fresh `helpers: {...}` literal and overwrite whatever the caller passed**. So forwarding `helpers` at the handler call site is a no-op for those pages — the wrapper's own literal is what reaches the template. The companion wrapper's literal was missing `buildAdminLocation`, which the template needs indirectly via `renderSubnav` (shared.js) and `renderBehaviourFormFields` — so EVERY companion tab (incl. identity) threw `buildAdminLocation is not a function` and the page came back blank, even after `extractRuntimeSettings` was added. Compare a broken wrapper against the sibling `renderBehaviourPage` wrapper, which already lists the full set (`extractRuntimeSettings, escapeHtml, withThemeField, renderOptions, buildAdminLocation`).

**How to apply:** when an admin page works in the standalone `preview.js` but is blank/broken on the deployed bot, diff the two helper builders and check the handler in `src/http/adminPageHandlers/` actually forwards `helpers`. Fastest verification: a tiny node harness that calls the render fn with `buildAdminPageHelpers(...)` output as `helpers` across all tabs — it surfaces missing keys immediately without needing a DB or Discord token.

**Separate trap — action handlers must pass `context` to the `withAdmin` wrapper.** Each POST action module returns `withAdmin(async (innerReq, innerRes, innerContext) => {...})(req, res, context)`. `withAdmin` immediately reads `context.config.admin`, so calling the wrapper as `(req, res)` (omitting `context`) throws synchronously → a 500 "Internal server error" on save (it isn't caught by the inner try/catch, which only wraps the handler body). This hit `inner-life-*` and `continuity-*` (every other action module passes `context`). Fast scan: `rg '\}\)\(req, res\);' src/http` should return nothing.

**Separate trap — upsert-by-id "update" branches are dead code without an Edit affordance.** The admin CRUD panels are server-rendered add/delete forms. A storage `upsert*({ id?, ... })` whose update branch keys on `id` is unreachable unless the UI surfaces an edit path: a per-row Edit link that sets a query param (e.g. `?editScheduleId=<id>`), the page **handler** reading that param and resolving it against the already-loaded list into an `editing` object, and the render fn prefilling the form (input `value=`, select `current=`, checkbox checked) plus emitting a hidden `id` field and an "Update" button. Resolve the editing row in the handler (not by echoing raw query content) and `escapeHtml` prefilled values / `encodeURIComponent` the id in the link. Without all three layers (link → handler lookup → hidden id in save action), "edit" silently behaves as "add a duplicate".

**Separate trap — unreachable sections fall through to the home page.** The full-bot dispatcher (`handleAdminPageRequest`) decides which page to render purely from `getAdminRouteState(pathname)` in `src/http/adminPageHandlers/shared.js`, which ends in a `return { section: "home" }` fallback. If a section's handler exists and is wired (`if (route.section === "X")`) but `getAdminRouteState` has **no case** for that path, the URL silently resolves to `home` and the handler is dead code — the nav link renders the home page instead of the section. This bit `continuity` and `inner-life`: handlers existed but `getAdminRouteState` had no path cases, so both nav links showed home on the deploy. Note the section *string* must match the dispatcher's check exactly (e.g. `innerLife` camelCase, not `inner-life`). When auditing routes: a faithful dispatcher harness that prints the resolved handler (a `/admin/continuity` throw landing in `handleHomePageRequest` is the tell) catches this; a per-handler harness will NOT, because it bypasses routing.
